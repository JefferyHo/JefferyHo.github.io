上一篇文章讨论了一个问题：前端工程师会不会被 AI Agent 替代。继续往下看，更实际的问题其实是，前端应该如何和 Agent 一起工作，以及前端相对于其他工种到底有什么优势和短板。

我的判断是，前端会较早感受到 Agent 的冲击，也会较早获得 Agent 带来的效率提升。页面结果看得见，反馈周期短，代码模式相对成熟，这些特点都适合自动化。但前端离用户最近，体验问题又很难完全用代码检查，所以前端不能只把 Agent 当成一个更快的代码生成器。

## 前端为什么适合和 Agent 协作

前端有几个天然优势。

首先，反馈速度快。改动一个按钮、表单或页面布局，通常可以马上在浏览器里验证。工程师很容易形成“修改、运行、观察、调整”的循环，这比需要等待大量数据积累的推荐、风控或运营系统更适合快速试错。

其次，前端的结果比较直观。页面结构、交互路径和视觉状态都可以直接展示给产品、设计和用户。Agent 生成的结果是否偏离目标，很多时候在第一次运行时就能被发现。

再次，前端有大量可复用的模式。表单、列表、弹窗、分页、加载状态、错误提示和权限控制都有成熟做法。Agent 可以快速完成初稿，工程师再根据项目约束进行调整。

最后，前端连接了产品、设计和技术。前端工程师既要理解用户流程，也要处理接口、状态、性能和兼容性。这个位置让前端有机会把模糊的想法快速做成可交互的原型，也能更早发现需求和实际使用之间的差距。

这些优势让前端很适合使用 Agent，但也带来一个风险：越容易描述和复用的工作，越容易被自动化。

## 不要把协作理解成“把需求发给 Agent”

Agent 能不能产出好结果，往往取决于工程师提供了多少有效上下文。只说“帮我做一个后台页面”，Agent 可能会生成一个看起来完整、实际上无法接入现有业务的页面。

一个可执行的任务至少要包含以下内容：

- 背景：页面服务谁，解决什么问题
- 范围：本次要完成哪些功能，不做哪些功能
- 约束：使用什么框架、组件库、接口和目录结构
- 状态：加载、空数据、错误、无权限和移动端如何表现
- 验收：需要通过哪些交互、测试和兼容性检查

例如：

```text
背景：这是运营人员查看订单异常的页面。
目标：新增筛选、分页和批量标记已处理功能。
约束：沿用现有 Table、Form 和权限判断，不新增依赖。
数据：使用 src/api/orders.ts 中的接口，先阅读类型定义。
验收：覆盖加载、空数据、请求失败、无权限和重复提交状态。
流程：先给出实现计划和风险，再修改代码，最后运行相关测试。
```

这不是提示词技巧，而是工程师把隐含判断显式写出来。上下文越清楚，Agent 的输出越容易检查，也越不容易在错误方向上快速前进。

## 一套比较稳妥的协作流程

### 第一步：让 Agent 先阅读和提问

不要一开始就让 Agent 改代码。先让它阅读相关目录、组件、接口和测试，然后回答几个问题：当前实现是什么，需求涉及哪些文件，可能影响哪些状态，哪些地方需要确认。

如果 Agent 没有提出任何疑问，反而直接生成了大量代码，也要保持警惕。真实项目通常不会简单到没有边界条件。

### 第二步：先要方案，再要实现

让 Agent 先给出实现计划、文件变更范围和风险。工程师可以在这一阶段纠正方向，成本比代码写完后再返工低得多。

计划不需要很长，但应该说明数据如何流动，状态如何变化，异常如何处理，以及如何验证结果。对于跨页面或跨服务的需求，还要确认接口契约和权限边界。

### 第三步：把任务拆成小的提交单元

不要一次让 Agent 完成整个功能。更合适的顺序通常是：

1. 阅读现有代码并补充计划
2. 实现数据类型和接口适配
3. 实现主要交互和状态管理
4. 补充加载、错误、空数据和无权限状态
5. 编写测试并更新文档
6. 检查性能、可访问性和移动端表现

每一步都应该可以独立检查。这样既方便定位问题，也能让工程师持续掌握关键决策。

### 第四步：让 Agent 自查，但不要让它自己验收

Agent 可以运行测试、检查类型、寻找重复代码和生成测试报告。这些工作很有价值，但它不能独立决定“这个功能对用户来说是否合理”。

工程师必须亲自确认交互是否自然，错误提示是否有帮助，权限是否正确，数据是否会泄露，以及页面在真实设备上是否稳定。代码通过检查，只说明它满足了部分机器规则。

一个清晰的责任分工是：

| 工作 | Agent 更适合负责 | 工程师必须负责 |
| --- | --- | --- |
| 代码实现 | 初稿、重复代码、类型和测试样例 | 架构取舍、关键逻辑和最终合并 |
| 需求理解 | 整理信息、列出疑问 | 确认目标、范围和优先级 |
| 质量检查 | 编译、静态检查、自动化测试 | 用户体验、风险和上线判断 |
| 知识积累 | 生成文档、总结变更 | 建立项目规范和长期约束 |

协作的目标不是让 Agent 代替工程师做决定，而是让工程师把更多时间放在需要判断的事情上。

## 前端相对于其他工种的优势

### 相对于后端，前端离用户更近

后端系统通常更重视数据一致性、吞吐量、事务和稳定性。前端则直接承接用户的行为和反馈。前端工程师更容易看到功能是否真的被理解、是否容易操作、是否在错误时给出了恢复路径。

这让前端在 AI 产品中有天然优势。模型输出、工具调用和任务进度最终都要通过界面呈现。没有清晰的状态设计，模型能力越强，用户越容易感到混乱。

前端的短板也很明显。用户看不到后端复杂度，团队有时会误以为页面只是“接个接口”。实际上，一个小功能可能同时涉及权限、缓存、竞态、网络失败、浏览器差异和无障碍支持。前端问题经常外表简单，内部边界很多。

### 相对于设计，前端更接近可执行结果

设计师负责探索体验、视觉表达和交互方案，前端负责把方案变成真实可运行的产品。前端可以直接验证动画是否流畅、表单是否能完成任务、错误状态是否可恢复。

但前端也容易把能实现误认为值得实现。一个视觉上漂亮的页面，不一定能降低用户的理解成本。和设计协作时，前端应该尽早讨论状态、内容、响应式布局和异常情况，而不是等设计稿完成后才开始切图。

### 相对于产品，前端更容易快速验证想法

产品经理通常从用户问题、业务目标和优先级出发，前端可以把这些内容快速变成可操作的原型。Agent 进一步降低了原型成本，让一个下午内完成多个方案成为可能。

但原型成本下降，也可能导致团队同时堆积很多没有验证过的功能。前端不能因为做得快，就默认需求值得上线。更好的做法是用低成本原型验证关键假设，再决定是否投入完整工程成本。

### 相对于测试，前端更容易发现体验问题

测试工程师擅长建立系统的验证范围，前端工程师则更熟悉交互实现和状态变化。两者都需要，但关注点不同。前端不应该把测试理解为上线前最后一步，而应该在设计状态和实现功能时就考虑可测试性。

前端的弊端是容易过度依赖“自己点过一遍”。手工验证有价值，却不能替代自动化测试、真实设备检查和异常场景覆盖。

## 前端最明显的几个弊端

### 重复性工作最容易被商品化

官网、表单、列表、管理后台和常见组件都有成熟范式，需求也容易转成代码。Agent 可以快速完成这些工作，因此只会组件拼装和页面复刻的能力，价值会下降得很快。

### 质量的一部分很难量化

视觉层级是否清楚，错误提示是否能帮助用户恢复，动画是否打扰操作，这些问题不能只靠编译器和单元测试判断。前端需要同时具备技术判断、产品意识和用户视角。

### 工具变化会掩盖基础薄弱

框架、组件库和 AI 工具都在快速变化。如果只追逐 API，工程师很容易积累一堆短期知识，却没有真正理解浏览器、网络、渲染、状态和工程质量。工具越强，基础越能决定一个人能不能发现工具的错误。

### 前端工作容易被切碎

一个页面可能由产品、设计、前端、后端、测试和运营共同完成。前端常常承担大量协调工作，却不一定拥有完整的决策权。使用 Agent 后，代码实现更快，但上下文同步、接口确认和验收责任不会自动消失。

因此，前端的优势是反馈快、离用户近、连接面广；弊端是重复工作容易被自动化，质量边界复杂，工作又容易被拆散。前端工程师要主动扩大自己的影响范围，否则很容易只剩下执行明确指令的部分。

## 不同阶段的实践建议

初级前端可以先让 Agent 帮自己阅读代码、解释浏览器行为、补充测试和完成小型功能，但每次都要亲自复盘生成结果。不要只复制代码，要能说清楚它为什么这样工作。

中级前端应该建立自己的协作模板：任务背景、约束、验收标准、测试命令和回滚方式。还要逐步掌握 API、数据库、鉴权、部署和日志，这样才能判断 Agent 提出的方案是否能真正上线。

高级前端需要把重点放到团队级问题上，例如代码库如何让 Agent 更容易理解，设计系统如何减少重复实现，测试和文档如何形成反馈，哪些任务适合自动化，哪些决策必须保留人工审查。

不要把提示词工程当成唯一竞争力。稳定的能力仍然是技术基本功、产品判断、系统设计、沟通协作和质量责任。Agent 会改变写代码的方式，但不会替工程师承担结果。

## 结语

前端和 AI Agent 的关系，不是“谁替代谁”，而是工作分工重新划定。Agent 擅长快速探索、实现和整理，前端工程师擅长理解用户、判断取舍、连接团队并对体验负责。

页面实现会越来越便宜。真正稀缺的，是知道什么问题值得解决，能把模糊需求变成可靠方案，并且在上线后继续观察和改进的人。
